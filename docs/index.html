PROJECT OVERVIEW

项目概览：一个可移植的 UE 运行时插件

ObjectPoolRuntime 是一个独立于具体玩法的 Unreal Engine 运行时插件，为 Actor、普通 UObject 和 ActorComponent 提供统一的池化能力。插件核心由世界级子系统、按类分桶容器、数据驱动配置、生命周期接口和网络复制支持构成。

CHAPTER 01

为什么需要对象池

在 UE 中，频繁执行 SpawnActor / DestroyActor、NewObject / GC、组件注册与反注册，以及网络复制对象的建立和清理，都会形成运行时成本。对象池把“每次创建与销毁”改造成“预先创建、用后复位、下次重新初始化”。

第一次预创建一批对象，承担一次性构建成本。
使用完不真正销毁，重置状态后放回池。
再次需要直接从池中取得并重新初始化。
主要收益减少分配释放、降低 GC 压力和战斗期突发 Spawn 卡顿。

这种模式尤其适用于高频召唤的 Actor、复制子对象和动态组件。它的价值并非让创建“免费”，而是把不稳定的尖峰工作前移，并让复用路径可控。

CHAPTER 02

总体架构：四层协作

  1. 配置层：决定哪些类能入池，编辑器、服务端、客户端分别预分配多少，以及采用哪种恢复策略。
  2. 管理层：UObjectPoolSubsystem 绑定 World 生命周期，负责预热和统一的获取/释放 API。
  3. 容器层：FPooledObjectBucket 按类保存不活跃对象，并记录容量与统计数据。
  4. 业务协作层：框架调用生命周期回调，对象自己清理和恢复业务状态。

公开入口包括 SpawnActorFromPool、GetObjectFromPool、GetComponentFromPool，以及三种对应的 Release*ToPool。Actor 的获取与激活统一由 SpawnActorFromPool 完成，同时建立 Transform、Owner、Instigator 和 ModeTag 组成的复用上下文。

CHAPTER 03

配置层：由模式与数据驱动

当前入口是 UObjectPoolSettings。它提供 ModeConfigs（FGameplayTag → FPoolModeConfig）、无需 Tag 的 DefaultModeConfig、DefaultModeTag、模式提供函数名 ModeTagProviderFunctionName 与全局开关 bEnabled。

每个模式配置都包含 Actor、UObject、Component 三组类配置。共同字段是：是否启用、Editor/Server/Client 预热数量、最大容量、是否允许运行时扩容，以及恢复策略。对象和组件还额外包含复制开关及远端副本释放策略。

为什么按 GameplayTag 切换

池容量和可池化类通常与游戏模式强相关。与其维护一份全局固定配置，不如让 GameState 的约定函数返回当前模式 Tag，再由池选择对应配置。解析失败时回退到默认 Tag 或 DefaultModeConfig。当前插件还提供 SwitchModeTag 与 RefreshModeTag；但只有在没有 Active 或过渡中条目时才允许切换，避免旧模式对象与新配置混杂。

恢复策略的权衡

当前策略框架行为业务责任
BusinessCallbacksOnly只执行业务回调；必要的所有权转移清理仍会发生最大，最快但最容易遗漏
ResetFrameworkState额外清计时器，并重置框架拥有的 Tick/激活/注册状态仍须清 Blueprint 变量、引用和技能状态
Full进一步清 Latent Action、附件等框架状态仍无法通用清材质、特效、物理速度、Buff、委托等

恢复越彻底越安全，但成本也越高；恢复越轻量，业务类越必须可靠地清理自身状态。

CHAPTER 04

管理层：为什么挂在 WorldSubsystem

对象池天然与 World 生命周期一致：地图加载时创建，地图卸载时销毁，不同 World 各自维护自己的池。DoesSupportWorldType() 限制支持 Game 与 PIE，因此它是运行时世界池，而不是全局单例池。

子系统分别维护 Actor、UObject 与 Component 的桶和 Managed 记录。每个 FPooledObjectBucket 对应一个类，并携带条目状态、最大容量、命中/未命中、创建量和预热量等运行数据。

按类分桶　key 是类，value 是该类的一组可复用对象。这是简单、可预测、便于配置与统计的结构。
CHAPTER 05

容器层：从简单数组到受约束状态机

FPooledObjectBucket 保存某个类的不活跃对象，避免重复入池，并在取出时跳过失效引用。按类小规模池结构直接、可预测；大量对象场景则需要结合命中率、容量和业务分布评估。

每个受管对象都遵循 Unmanaged → Preallocating → Active → Releasing → Inactive 状态约束。Preallocating 与 Releasing 是受保护的过渡态，递归释放会被拒绝；生命周期回调若让对象失效，系统会重新验证所有权与桶状态后再继续。

CHAPTER 06

生命周期：Initialize、BeginPlay 与异步预热

Initialize() 建立 World 生命周期联系；池实例创建后不会立即预分配，而是在 World 真正进入 BeginPlay 时统一预热。插件首先检查设置中的 bEnabled，随后依次启动 Actor、UObject 与 Component 的预热流程。

Initialize→OnWorldBeginPlay→ResolveCurrentModeTag→异步加载 SoftClass→按端选择数量→创建并进入 Inactive

三类预热的共同套路是：解析当前模式、收集启用类的 SoftClassPath、异步加载、按 Editor/Server/Client 数量创建，然后通过统一的释放/停用路径让对象以一致状态进入池。当前实现还把目标不活跃数量限制在 0..MaxPoolSize。

三种对象怎样预创建

  • Actor：通过标准 Actor 创建链预热，而不是裸 NewObject<AActor>，以保证组件、蓝图与构造流程接近正常 Spawn。
  • UObject：NewObject 后移入池专用 Outer，并进入不活跃状态。
  • Component：创建动态组件并建立完整状态，入池时保持反注册并移动到组件池 Owner；再次分配时即可安全转交给另一个 Actor。
CHAPTER 07

取出：Actor、UObject、Component 各有不同契约

Actor

SpawnActorFromPool 会先从桶中取得不活跃 Actor；若没有命中且配置允许扩容，则创建新 Actor。随后构造 FActorPoolAcquireContext，填入 Transform、Owner、Instigator 与当前 ModeTag，并根据权威端/客户端派发 OnPoolAcquireServer 或 OnPoolAcquireClient。池负责框架流程，真正的业务复位、位置朝向和启用逻辑由 Actor 回调完成。

UObject

GetObjectFromPool(Outer, Class) 取得对象后，把它从池专用 Outer 迁移到新的 Outer，再调用 OnObjectAcquired。Outer 决定生命周期归属、对象路径和部分查找关系；如果不迁移，复用对象可能仍挂在旧拥有者下。

ActorComponent

GetComponentFromPool(Owner, Class) 取得组件后，先规范化旧注册和旧世界状态，再转交给新 Owner 并调用 OnComponentAcquired。取出阶段仍可能执行反注册，这是为了先清理旧挂接，再由新的业务流程完成注册与激活。

CHAPTER 08

回收：先结束身份，再进入容器

Actor 回收

ReleaseActorToPool 验证目标确为当前子系统管理的 Active Actor，使其进入 Releasing，按恢复策略停用，并派发服务端或客户端 Release 回调，最后放回对应类桶。Actor 先在业务回调中清理自己，池负责存放、容量与状态约束。

UObject 回收

系统验证配置与管理关系，执行复制注册和远端副本策略，调用 OnObjectReleased，再将对象移回池专用 Outer 与对象桶。若网络策略无法保证远端身份已结束，则当前服务端实例不会被重新使用。

Component 回收

除复制身份外，还必须处理注册与 Owner：组件从旧 Actor 反注册，按策略清理框架状态，调用 OnComponentReleased，移交给组件池 Owner，最后进入对应类桶。

回收入口既可由游戏逻辑主动调用，也可接入 Actor 销毁、技能实例清理或网络 Channel 清理等生命周期节点；接入方必须保证同一对象不会重复或递归释放。

CHAPTER 09

网络复制：内存复用不等于网络身份可复用

网络对象回收必须结束其旧的复制身份，包括停止后续复制、清除 registered subobject 关系、处理远端副本，并解除旧 Owner 与 Channel 上下文。否则对象换到新的 Owner 或业务上下文后，可能出现网络身份冲突、Replicator 残留或客户端状态错乱。

Actor 的复制生命周期由 UPooledActorNetworkStateComponent 协调；UObject 与 Component 则由子系统根据远端释放策略决定能否保留服务端实例。这里不能只把“本地已放回数组”当成“网络上已经安全回收”。

当前插件的精确契约

  • 池化 Actor 使用一个复制生命周期状态：Active、Generation、AcquireContext。Generation 改变可在 Release/Acquire 被合并进一次复制更新时重建 Release 回调。
  • 复制 UObject 必须由权威、可复制 Actor 通过 registered subobject list 持有。
  • 复制组件在所有权转移前必须从旧 Actor 反注册。
  • DestroyRemoteReplica：显式销毁远端副本，服务端同一实例才可安全回池复用。
  • UnregisterOnly：只停止后续复制，不保证现有客户端副本立刻消失；为避免身份/Owner 冲突，服务端实例被丢弃而不复用。
  • TearOffRemoteReplica：留下独立客户端副本；被 TearOff 的网络身份不能安全复用，因此服务端实例同样丢弃。
CHAPTER 10

GC 与业务边界：框架不能替对象理解自己

池中对象不一定在场景中活跃，也未必位于普通 UObject 引用链上。插件通过子系统拥有的池 Outer、组件池 Owner、容器与 Managed 记录配合 UE 引用体系，维持闲置对象的可达性，避免仍在池中的对象被 GC。

从正常生命周期抽离对象之后，框架必须明确告诉 GC：“对象虽然闲置，但还不能回收。”但它无法通用恢复所有业务状态，例如：

运行状态Tick、可见性、碰撞、激活、物理速度
异步与事件绑定事件、计时器、Latent Action、动态委托
所有权与引用Owner、Instigator、Outer、子对象、目标引用
游戏语义蓝图变量、技能、Buff、材质、Niagara、音频

边界必须清楚：框架负责预分配、存放、取出、容量、GC 保护、所有权迁移和网络复制层处理；业务对象负责如何回收自己、如何重新启用自己。Actor 回调是 OnPoolReleaseServer/Client 与 OnPoolAcquireServer/Client，UObject/Component 则是 OnObjectReleased/Acquired 与 OnComponentReleased/Acquired。

CHAPTER 11

三条完整主流程

Actor

BeginPlay / Prewarm→标准 Spawn→ReleaseActorToPool→Inactive→SpawnActorFromPool→Acquire 回调

预热阶段由子系统解析模式、异步加载类并按当前运行端创建 Actor；再次使用时从对应桶取出，构造 FActorPoolAcquireContext，派发服务端或客户端 Acquire 回调，由 Actor 恢复业务状态。

UObject

NewObject→ReleaseObjectToPool→复制策略 + Released→池 Outer→GetObjectFromPool→新 Outer + Acquired

回收时结束复制关系、调用 Released 回调并迁移到池 Outer；复用时迁移到新的 Outer，再调用 Acquired 回调。

ActorComponent

创建组件→建立完整组件状态→ReleaseComponentToPool→Unregister + 池 Owner→GetComponentFromPool→新 Owner + Acquired

回收时按策略结束复制关系、反注册并转移到组件池 Owner；取出时先规范化旧状态，再迁移到新 Owner 并触发 Acquired 回调。

CHAPTER 12

诊断、边界与关键结论

  • 不要把当前主方案与旧的 FObjectRecyclePool / RecycleSet 混在一起。
  • 配置按模式 Tag 切换，不是一份写死的全局配置；简单项目也能使用 DefaultModeConfig。
  • 这不是纯 Actor 池，它同时支持普通 UObject 和 UActorComponent。
  • 网络复制清理是核心难点，尤其是跨 Owner 复用的子对象和组件。
  • 框架只重置它拥有的状态，任意业务变量仍必须由生命周期回调清理。
  • 所有获取、释放、预热与模式切换 API 都只能在 Game Thread 调用。

当前项目自带的观测工具

Pool.Dump 输出池数量、命中率与预热状态；Pool.Validate 检查 Bucket→Managed 和 Managed→Bucket 所有权不变量；Pool.ValidateOnMutation 1 可在每次获取/释放后检查；Pool.NetSmoke 验证 Actor 复用以及 UObject/Component 跨 Owner 复用，并可用 Pool.NetSmoke.ReleasePolicy 0|1|2 选择三种远端释放策略。

最终结论　对象池真正复用的不是一块孤立内存，而是一个曾经属于某个 World、Owner、GC 引用链和网络身份的对象。安全复用的核心，是让旧身份完整结束，再让新身份按明确契约开始。
CHAPTER 13

接入实战：从安装到验收

这一章把前面的设计契约落到实际项目中。目标不是“能从池里拿出对象”就算完成，而是让对象在多次获取、释放、跨 Owner 以及网络环境下都保持一致。

第一步：安装与启用插件

将 ObjectPoolRuntime 仓库放入目标项目的 Plugins/ObjectPoolRuntime。插件目录下应直接看到 ObjectPoolRuntime.uplugin 和 Source。
重新生成项目文件并编译 Editor Target。插件依赖 Core、CoreUObject、Engine、DeveloperSettings、GameplayTags 等 UE 模块，不依赖 Shooter 或 Horror 示例玩法。
在编辑器的 Plugins 窗口确认 Object Pool Runtime 已启用，然后在 Project Settings 中找到 Object Pool 配置页。
第一次接入只配置一个低风险类，验证完整生命周期后再逐步扩展，不要一次把所有高频对象都改成池化。

第二步：理解每个配置字段

字段作用建议
bEnabled决定该类是否参与池化调试单个类时可快速关闭，不必删除配置
EditorPreallocateCount编辑器运行环境的预热数量用于 PIE 调试，通常接近客户端或略小
ServerPreallocateCount权威服务器预热数量按服务器同时活跃峰值与生成速率估算
ClientPreallocateCount客户端预热数量只覆盖客户端实际会创建或持有的实例
MaxPoolSize每个类桶的容量上限必须大于等于预热数，并留出合理峰值余量
bAllowRuntimeGrowth池未命中时是否允许创建新实例稳定性优先时开启；严格预算场景可关闭并监控 Miss
RecoveryPolicy框架清理强度先用 ResetFrameworkState，确认回调完整后再按性能调整
配置校验　插件启动时会调用 ValidateSettings。负数预热数量、预热数超过最大容量、同一模式重复配置同一个类，以及无效类路径都应在进入游戏前修正。

第三步：实现 Actor 生命周期

池化 Actor 实现 IPoolableActorInterface。OnPoolCreated 只在实例首次创建时调用，适合做不会随每次使用变化的初始化；Acquire/Release 回调则必须支持多次执行。

void AMyProjectile::OnPoolAcquireServer_Implementation(
    const FActorPoolAcquireContext& Context)
{
    SetActorTransform(Context.Transform);
    SetOwner(Context.Owner);
    SetInstigator(Context.Instigator);
    SetActorHiddenInGame(false);
    SetActorEnableCollision(true);
    SetActorTickEnabled(true);
    RemainingLife = MaxLife;
}

void AMyProjectile::OnPoolReleaseServer_Implementation()
{
    GetWorldTimerManager().ClearAllTimersForObject(this);
    SetActorEnableCollision(false);
    SetActorTickEnabled(false);
    SetActorHiddenInGame(true);
    SetOwner(nullptr);
    SetInstigator(nullptr);
    HitActors.Reset();
}

客户端回调应恢复纯客户端表现，例如可见性、特效或插值状态；不要在客户端回调中执行只有服务器才有权决定的伤害、生成或持久状态变更。

第四步：实现 UObject 与 Component 生命周期

UObjectOnObjectPoolCreated 做一次性初始化；OnObjectAcquired 读取新的 Outer/Owner 上下文并恢复状态；OnObjectReleased 解绑委托、清目标引用、取消异步任务和业务计时。
ActorComponentOnComponentPoolCreated 做一次性初始化；OnComponentAcquired 面向新 Owner 建立绑定；OnComponentReleased 停止 Tick、解绑旧 Owner、清物理/特效状态。
UObjectPoolSubsystem* Pool = World->GetSubsystem<UObjectPoolSubsystem>();

UObject* PooledObject = Pool->GetObjectFromPool(NewOuter, ObjectClass);
UActorComponent* PooledComponent =
    Pool->GetComponentFromPool(NewOwner, ComponentClass);

// 使用结束后必须调用与类型对应的释放入口。
Pool->ReleaseObjectToPool(PooledObject);
Pool->ReleaseComponentToPool(PooledComponent);

第五步：正确选择远端释放策略

DestroyRemoteReplica：真正的跨周期复用
服务端在回收前明确移除客户端副本，随后同一个服务端 UObject 或 Component 实例可以进入池并用于新的 Owner。它最符合传统对象池语义，但需要完整、可靠的远端销毁路径。
UnregisterOnly：停止复制，但不承诺远端立即消失
系统只停止后续复制。客户端旧副本可能继续存在一段时间，因此服务端实例不能换 Owner 后再次复用，否则会产生网络身份冲突。该实例会被丢弃。
TearOffRemoteReplica：保留独立客户端副本
客户端副本脱离服务器继续存在，适合希望远端表现自然结束的场景。由于旧网络身份被保留，服务端实例同样不能重新进入池。

第六步：容量规划与命中率

预热数量不应该凭感觉填写。先记录一个代表性战斗或关卡窗口内的同时活跃峰值、每秒生成峰值、平均存活时间和突发系数。初始预热量可接近稳定期 P95 活跃数，MaxPoolSize 再覆盖短时峰值。

命中率低预热不足、容量太小、对象存活过长，或类配置没有命中当前 ModeTag。
池长期过大预热数或 MaxPoolSize 过高，模式切分不够细，闲置对象占用内存。
Miss 但不扩容bAllowRuntimeGrowth=false，调用方必须接受获取失败并准备降级逻辑。
创建数持续增长检查对象是否确实释放、释放是否被容量上限拒绝，以及远端策略是否要求丢弃实例。

第七步：常见问题定位

现象优先检查典型原因
第二次使用带着旧目标Release 回调业务引用、委托、数组或 Blueprint 变量未清理
组件仍挂在旧 ActorOwner 转移与注册状态业务回调重新注册过早，或持有旧 Owner 引用
客户端没有收到 AcquireActor 网络状态组件与 GenerationActor 未正确复制，或生命周期状态没有进入复制路径
模式切换失败Active / Releasing 条目仍有对象正在使用或处于过渡态
池中对象被 GC是否由当前子系统管理绕过创建/获取 API，或对象已被回调主动销毁
Release 返回 false对象状态与归属对象无效、不是 Active、已在池中、容量已满或不属于该子系统

第八步：发布前验收清单

  • 同一实例连续获取/释放至少 100 次，没有状态串扰。
  • Actor 的 Transform、Owner、Instigator、Tick、碰撞、可见性和计时器都能正确复位。
  • UObject 跨 Outer、Component 跨 Owner 后不保留旧引用或注册关系。
  • 服务端与客户端回调只执行各自应负责的逻辑。
  • 三种远端释放策略均通过 Pool.NetSmoke 验证。
  • Pool.Validate 没有 Bucket/Managed 所有权不一致。
  • Pool.Dump 的命中率、创建量与容量符合预期。
  • 地图卸载、PIE 重启和模式切换后没有残留对象或异步加载回调。
推荐调试组合　在开发环境临时开启 Pool.ValidateOnMutation 1，跑自动化测试和网络烟测，再用 Pool.Dump 观察真实玩法中的命中率。确认稳定后关闭逐次校验，避免把诊断成本带入常规运行。
